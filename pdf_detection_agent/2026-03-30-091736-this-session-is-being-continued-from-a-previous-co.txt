в—Ҹ Three ways, depending on what you're doing:

  1. Fix a wrong bbox (edit mode)
  Select the detection row вҶ’ click P1 on image вҶ’ click P2 вҶ’ auto-saved 
  immediately as confirm

  2. Mark a false positive
  Select the row вҶ’ set Judgement = "wrong" вҶ’ рҹ’ҫ Save Correction вҶ’ saved as
  reject

  3. Add a missed column
  Make sure NO row is selected вҶ’ click P1 вҶ’ click P2 вҶ’ set Shape вҶ’ вһ• Save New
  Detection вҶ’ saved as add

  All three call agent.add_correction(tile_hash, action, shape, bbox_norm) where
   bbox_norm is the bbox as 0вҖ“1 floats relative to tile size. These get injected
   into the moondream prompt on the next detection run as:

  Prior human corrections вҖ” REAL columns: square@[0.12,0.34,0.18,0.41] | FALSE
  positives to skip: round@[0.5,0.5,0.55,0.6]

  tile_hash is set. The hash ties your corrections to the exact tile content вҖ”
  if you save without a hash, nothing gets stored.

вқҜ why all llava-* cannot use here?                                              
в—Ҹ The crash comes down to one specific component: the LLaVA MMPROJ file (the
  multimodal projector that bridges vision and language).

  Here's what happens when you send an image to any llava-* model:

  Your image
      вҶ“
  CLIP vision encoder  вҶҗ runs on GPU вҶҗ CRASHES HERE
      вҶ“
  MMPROJ projector (converts image patches to token embeddings)
      вҶ“
  LLaMA language model (generates text)

  The CLIP encoder + projector (sha256-eb569aba...) is the shared component
  across ALL your llava models вҖ” llava:7b, llava:7b-v1.6, llava-llama3, and
  florencemed all use the exact same MMPROJ blob. When Ollama launches the model
   runner and feeds an image through the CLIP encoder on your GPU, the runner
  process crashes (segfault or illegal CUDA instruction).

  Why moondream and qwen work:
  - moondream вҖ” uses phi-2 architecture with CLIP embedded inside the main GGUF,
   processed differently
  - qwen2.5vl вҖ” uses Qwen's own SigLIP vision encoder, completely different CUDA
   code path

  Why CPU works for llava:
  - With num_gpu=0, the CLIP encoder runs in pure C++ without any CUDA calls вҶ’
  no crash

  Root cause candidates:
  1. Your GPU's CUDA compute capability is incompatible with CUDA ops used in
  that specific CLIP GGUF build (the Ollama binary ships its own llama.cpp with
  specific CUDA arch targets)
  2. A CUDA kernel in the old LLaVA projector format uses an instruction not
  supported by your GPU generation

  Check your GPU:
  nvidia-smi --query-gpu=name,driver_version,compute_cap --format=csv,noheader

  If you're on an older GPU (compute capability < 7.5) or a very new one not yet
   in Ollama's target list, that's the mismatch. The fix would be to rebuild
  Ollama from source targeting your exact GPU arch, or wait for a newer Ollama
  release that updates the llama.cpp CUDA kernels.
