---                                                                             
                                                                                
## 1. Executive Objective                                                       
Transform the legacy algorithmic PDF-to-BIM logic found in                      
`~/Documents/mcc-amplify-ai` into a robust, self-correcting multi-agent system  
within `~/Documents/mcc-amplify-v2`. The system must learn from Revit API       
failures and geometric mismatches to improve accuracy over time.                
                                                                                
## 2. Agent Architecture                                                        
The workflow is divided into three specialized agents, coordinated by a central 
 controller:                                                                    
                                                                                
| Agent | Responsibility | Key Tools |                                          
| :--- | :--- | :--- |                                                          
| **Vision-Analyst** | Interprets PDF/Image segments via DeepLSP or YOLO v11. | 
 `pdf_processor`, `yolo_detector` |                                             
| **Validation (The Harness)** | Runs "Architectural Sanity Checks" (closed     
loops, DfMA wall thickness). | `geometry_validator` |                           
| **BIM-Translator** | Converts validated geometry into C#/.NET JSON for Revit. 
 | `revit_api_interface` |                                                      
                                                                                
---                                                                             
                                                                                
## 3. Technical Infrastructure                                                  
                                                                                
### A. Centralized Tooling (`tools.py`)                                         
All agents must interact with the environment through a unified toolset:        
* **`pdf_processor`**: Extraction of vector/raster data from specific           
coordinates.                                                                    
* **`geometry_validator`**: Rule-based engine to check for wall overlaps and    
structural grid alignment.                                                      
* **`revit_api_interface`**: Sends payloads to the "Revit Model Builder" and    
captures C#/.NET exception logs for feedback loops.                             
* **`memory_interface`**: Standardized CRUD operations for the persistent       
database.                                                                       
                                                                                
### B. Persistent Memory Layer (`memory.sqlite` or `memory.json`)               
Agents must utilize a "Lessons Learned" schema to prevent repeat errors:        
* **`task_id`**: Unique ID for the floor plan segment.                          
* **`input_features`**: Metadata (e.g., "Hand-drawn style," "High noise,"       
"Metric/Imperial").                                                             
* **`agent_reasoning`**: The internal "Thought" process leading to the action.  
* **`outcome`**: Success/Failure status + specific Revit Error Codes.           
* **`correction`**: The specific adjustment that resolved a prior failure.      
                                                                                
---                                                                             
                                                                                
## 4. Engineering Protocols                                                     
                                                                                
### Harness Engineering (The Reliability Layer)                                 
* **Self-Correction Loop:** If the `geometry_validator` or Revit API returns an 
 error, the agent must:                                                         
    1. Log the failure to Memory.                                               
    2. Query Memory for similar historical fixes.                               
    3. Re-reason and generate a corrected geometry payload.                     
* **Gold Standard Testing:** Use snippets from `mcc-amplify-ai` as "Ground      
Truth" to benchmark agent performance.                                          
                                                                                
### Context Engineering (The Data Management Layer)                             
* **Dynamic Context Window:** Instead of a full PDF, pass agents a "Context     
Package" containing:                                                            
    * Local tile coordinates.                                                   
    * Relevant Legend data (Wall types/Symbols).                                
    * Neighboring geometry (to ensure wall-to-wall connectivity).               
                                                                                
### Prompt Engineering (The Logic Layer)                                        
* System prompts must explicitly instruct agents to: *"Always check memory for  
previous failures on similar floor plan layouts before proposing a geometry."*  
* Use Few-Shot prompting for "Bad Detection" vs. "Corrected Detection."

‚ĚĮ # Engineering Directive: Agentic BIM Workflow Evolution (v2)                  
**Project:** MCC-Amplify-AI-v2                                                  
**Architecture:** Isolated Decentralized Agents                                 
**Context:** Main Contractor DfMA Integration (Singapore Standards)             
                                                                                
---                                                                             
                                                                                
## 1. Executive Objective                                                       
Transition from hard-coded PDF-to-BIM logic in `v1` to a self-correcting,       
decentralized agentic pipeline in `v2`. Each agent operates with **total        
autonomy**, possessing its own specialized toolset and private persistent       
memory to ensure high-fidelity Revit generation.                                
                                                                                
## 2. Decentralized Agent Architecture                                          
The workflow is orchestrated by a controller, but each agent is a               
self-contained unit:                                                            
                                                                                
| Agent | Status | Private Infrastructure | Responsibility |                    
| :--- | :--- | :--- | :--- |                                                   
| **PDF Detection** | **Ready** | `detection/tools.py`,                         
`detection/memory.sqlite` | Feature extraction from PDF segments. |             
| **Validation** | **New** | `validation/tools.py`, `validation/memory.sqlite`  
| DfMA & Architectural logic "Harness." |                                       
| **BIM-Translator** | **New** | `translator/tools.py`,                         
`translator/memory.sqlite` | C#/.NET Mapping for Revit API. |                   
                                                                                
---                                                                             
                                                                                
## 3. Isolated Persistent Infrastructure                                        
                                                                                
### A. Agent-Specific Tooling (`tools.py` per Agent)                            
**NO shared `tools.py` allowed.** Each agent directory must contain its own:    
* **Validation Agent Tools:** `geometry_checker`, `loop_closer`,                
`standard_thickness_lookup`.                                                    
* **BIM-Translator Tools:** `revit_schema_mapper`, `coordinate_transformer`,    
`revit_api_client`.                                                             
* **Memory Tools:** Localized `memory_io` functions specifically tuned for that 
 agent's data schema.                                                           
                                                                                
### B. Private Memory & Learning (`memory.sqlite` + `memory.json`)              
Each agent maintains its own private database to optimize its specific domain:  
* **Validation Memory:** Stores "Geometric Conflict Resolutions" (e.g., how it  
fixed a 145mm wall to a 150mm DfMA standard).                                   
* **Translator Memory:** Stores "API Success Patterns" (e.g., which C#          
parameter combinations successfully generated a complex curtain wall).          
* **Format:** Use `sqlite` for high-volume historical lookups and `memory.json` 
 for "Active Session" context or human-readable "Lessons Learned" logs.         
                                                                                
---                                                                             
                                                                                
## 4. Engineering Protocols                                                     
                                                                                
### Context Engineering (The Handoff)                                           
* **Payload Enrichment:** The controller acts as a courier, passing the output  
of Agent A (plus project metadata) into the input of Agent B.                   
* **Neighborhood Isolation:** To maintain agent focus, provide only the         
immediate geometric neighborhood required for the current task.                 
                                                                                
### Harness Engineering (Decentralized Feedback)                                
* **Local Self-Correction:** 1. If the **BIM-Translator** hits a Revit error,   
it records it in `translator/memory.sqlite`.                                    
    2. It attempts a fix using its own `translator/tools.py`.                   
    3. If it cannot fix the error, it sends a "Refinement Request" back to the  
**Validation Agent**, which then uses its own memory to propose a new geometric 
 solution.

  ---
  Running the Pipeline (Same Windows Machine)

  Your setup: Windows machine running both Revit and this codebase (via WSL2 or
  directly).

  Step 1 ‚ÄĒ Build & start the Revit C# service (Windows side)

  Open Command Prompt in revit_server/csharp_service/ and run:

  build.bat
  Then:
  run.bat

  This starts the Revit API service on http://localhost:5000. The config.json
  already has port 5000 and API key my-revit-key-2023.

  Step 2 ‚ÄĒ Set environment variables (Linux/WSL side)

  export REVIT_SERVER_API_KEY=my-revit-key-2023
  export WINDOWS_REVIT_SERVER=http://localhost:5000
  export REVIT_CLIENT_PATH=/path/to/mcc-amplify-ai   # so translator can find 
  RevitClient

  Step 3 ‚ÄĒ Run the frontend

  cd /home/jiezhi/Documents/mcc-amplify-v2/frontend
  ./run_frontend.sh

  This starts the web UI (Vite dev server). Open your browser at the URL it
  prints (typically http://localhost:5173).

  Step 4 ‚ÄĒ Run the pipeline directly (CLI)

  cd /home/jiezhi/Documents/mcc-amplify-v2
  python controller.py /path/to/floor_plan.pdf --page 0

  The .rvt file is written to data/models/rvt/<job_id>.rvt.

  Step 5 ‚ÄĒ Wire the frontend to the controller

  The frontend/run_frontend.sh starts the UI, but the frontend needs a backend 
  server to call controller.py. The revit_server/python_service/ directory
  likely has one ‚ÄĒ check its config.json for the host/port, then run it
  alongside the frontend.
